# Where Tool Effort Pays Off: A 20-Lemma Static-Replay Audit of the A/B/C Information Need at Each Proof Frontier

## 1. Executive Summary

Across 20 answer-proofs spanning easy → very_hard and 17 distinct kinds (game-hops, sampling equivs, ZK simulators, loop invariants, OW/DDH reductions, extractors, losslessness, pure induction), the real work decomposes cleanly into three classes:

- **A (白给 / read off the goal):** navigation openers, intro patterns, identity-coupling `rnd`, `proc; inline *`, trivial closers. **~20-25% of frontiers**, ~0% of the *difficulty*.
- **B (semantic / must invent):** loop/relational invariants, coupling bijections, the reduction idea, the case-split insight. **~25-30% of frontiers but ~90% of the difficulty.** Confirmed **1-3 per proof**, almost always tool-irreducible.
- **C (mechanical / must assemble):** which named lemmas to rewrite/apply, the structural navigation sequence, indices/variants. **~45-50% of frontiers and the dominant *repeated, gilable* cost** — i.e. the bulk that scales with proof length and that the agent grinds out turn after turn.

Counting frontiers across all 20 lemmas (≈180 classified frontiers): roughly **A ≈ 22%, B ≈ 28%, C ≈ 50%**. But B and C are not symmetric in nature. **B is a small fixed budget per proof (the 1-3 inventions) that does not grow much with length; C grows roughly linearly with proof length.** A long very_hard proof (Bad_bound, ~18 frontiers) still has only ~4 B-cores but a dozen C-frontiers of bigop/fmap/real-order assembly.

**The single highest-value lever is `operator_lemmas`** — live EC `search` over the loaded context (project-local included). It is the named tool target in **the C-class frontier of essentially every lemma**, and in 17 of 20 lemmas the audit explicitly calls it the "dominant repeated cost." The mechanical lemma-finding for bigop/fmap/real-order/distribution families is the largest single addressable slice of total work. The audit's calibration note (PIR) is corroborated 20×: "The mechanical lemma-finding (C) was the dominant, repeated cost."

A blunt way to state the verdict: **no tool can shrink B (the agent's job), A is already free, so all tool ROI lives in C — and operator_lemmas already captures the largest chunk of C.** The next lever (Section 5) is the *other* half of C that operator_lemmas does not touch: navigation indices, swap offsets, and the phoare/seq tactic-form scaffolds.

## 2. The Three Categories, Evidenced

### A — 白给 / Read Off The Goal (~22% of frontiers, ~0% of difficulty)

What consistently falls here, with cross-lemma evidence:

- **Procedure openers / inlining.** `byequiv => //; proc; inline *.` is A in `schnorr_proof_of_knowledge_shvzk`, `pedersen_computational_binding`, `es0_Gb`, `Gb_half`. The program text dictates every token.
- **Intro/destruct patterns read off the statement.** `move=> hf hfinv.` (D4_D6), `move=> Hneq Hr Hr'.` (schnorr special soundness), `move=> &1 &2 a.` (D4_D6) — "read straight off the lemma statement's premises."
- **Identity-coupling `rnd`.** When both sides sample the same distribution and need syntactic equality, `rnd` with no bijection is A: D4_Sample (`proc; rnd; skip=> />`), pedersen (`rnd; skip`), cpa_ddh0's bit coupling. The audit flags the recognition that *no nontrivial coupling is needed* as the only (marginal) content.
- **Trivial arithmetic closers.** `by skip; progress=> /#.` (Bad_bound), `smt().` chaining h1∧h2 (pr_Game0_Game1), `done` (pedersen), `=> //` (sxor2_cons).
- **Guard-mirroring case splits.** `case (is_restr x N)` (Pr_PIR_s) is A: read directly off the lemma's own RHS `if is_restr x N then ... else 0%r`.

A is genuinely free to the agent. **Surfacing it saves nothing**; a navigation menu that merely lists `proc; inline *` is offering the agent what it already sees.

### B — Semantic / Must Invent (1-3 per proof, tool-irreducible)

The audit confirms the calibration: **every proof has 1-3 B-cores, no more.** They fall into four recurring archetypes:

**(i) Relational / loop invariants that are NOT live-var equalities.** These are the irreducible inventions and the place where a "relate the live vars" skeleton actively *misleads*:
- `PIR_correct`: `big predT a PIR.s +^ big predT a PIR.s' = (if i0 < j then a i0 else zerow)` — "expresses WHY PIR is correct... not derivable from the program text by any tool."
- `Pr_PIR_s`: the membership-indicator invariant `forall k<j, ((k in s1)=(k in y)) = ((k in s2)=(k in x))` — "the two runs build DIFFERENT sets s1!=s2, so this is NOT a live-var equality and no relate-live-vars skeleton yields it."
- `Bad_bound`: the counting invariant `size P.logP = C.cP /\ C.cP <= qP /\ card(fdom F.m) <= C.cF /\ C.cF <= qF`.

**(ii) Sampling coupling bijections.** The heart of relational sampling proofs, never tool-derivable:
- `schnorr_..._shvzk`: `rnd (fun z => z - e{1}*w0{1}) (fun r0 => r0 + e1{2}*w0{2})` — the affine change-of-variables z = r + e·w "is the agent's irreducible invention and the whole point of the proof."
- `Pr_PIR_s`: `rnd (fun b => if (j{1} in y)=(j{1} in x) then b else !b)` — copy-or-flip the coin.
- `es0_Gb` / `D4_D6`: OTP xor-translation `w |-> w +^ (b?m1:m0)` and the bijection-collapse `finv r = a ⇔ r = f a`.

**(iii) The reduction / framing idea.** Choosing the intermediate game, the up-to-bad postcondition, or the library lemma to reuse:
- `pr_Game0_Game1` / `Plog_Psample`: the up-to-bad framing `res{1} => res{2} \/ Bad{2}` plus the `eq_except (pred1 r)` map-coupling.
- `Bad_bound`: the conditional-`sumid` triangular-sum bound — "the origin of the headline (qP-1)*qP/2 term... no tool can synthesize it."
- `G0_G1_G2`: recognizing `OnBound.ROM_BadCall` is the off-the-shelf bound and instantiating `r := true`.

**(iv) The escape/termination or case-split insight.**
- `Sample_lossless`: `while true (if test r then 1 else 0) 1 (mu sample (predC test))` — inventing the 0/1 variant *and* the escape-probability lower bound.
- Independence-deferral swaps that produce a 1/2: `swap 4 3` (fakecommit_half), `swap 4 4` (pr_Game2), `swap 4 4`... realizing the final coin is independent of the adversary view.

**Confirmation that B is tool-irreducible:** in all 20 lemmas the B frontiers carry `tool: none-agents-job` or, at most, `invariant-frame`/`tactic_forms` as a *scaffold* the audit explicitly downgrades to "marginal." Even the lightest B (sxor2_cons's `exists (j::s1) s2` witness map) is invention, not lookup.

## 3. Difficulty Scaling

The data answers the decisive question — *does difficulty add more semantic cores or more mechanical assembly?* — clearly:

| Difficulty | Lemmas | Typical #B-cores | Typical C-load | Frontiers |
|---|---|---|---|---|
| easy | sxor2_cons, fakecommit_half, Gb_half, pr_Game2 | 1-2 | thin (a few `*_ll` + one mass lemma) | 4-9 |
| medium | pedersen, BR93_correct, schnorr SS, D4_Sample, cpa_ddh0 | 1-2 | moderate (one lemma family) | 9-13 |
| hard | pr_Game0_Game1, D4_D6, pr_Game2_OW, G0_G1_G2, PIR_correct | 2-3 | heavy (multiple families + swaps) | 9-13 |
| very_hard | Bad_bound, Plog_Psample, schnorr SHVZK, Pr_PIR_s | 2-4 | very heavy (bigop + fmap + real-order chained) | 10-18 |

**Finding: difficulty scales C far more than B.**

- **B grows slowly and saturates.** Easy proofs have ~1 core (Gb_half: just `pred1 b'`); very_hard proofs have ~3-4 (Bad_bound: the sumid bound + counting invariant + seq schedule + body case-split). B roughly **doubles** easy→very_hard.
- **C explodes.** Easy proofs name ~3 lemmas (Gb_half: `Ag_ll`, `dt_ll`, `dbool1E`). Very_hard proofs name **10-12** and chain them within single frontiers (Bad_bound's conseq side-goal alone: `big_addn`, `big_split`, `big_constz`, `count_predT`, `size_range`, `ler_maxr`, `ler_wpmul2r`, `ler_add`, `ler_pmul`, `sumidE`). C grows **3-4×** easy→very_hard, and the *chaining depth per frontier* grows on top of that.

The qualitative signature confirms it. Easy proofs are "one independence insight + naming three lemmas." very_hard proofs are "one (or a few) genuine inventions **wrapped in a very large mechanical shell**" — the exact words for Bad_bound. The semantic kernel of Bad_bound (the triangular sum) is not harder to *state* than Gb_half's coin insight; what makes it very_hard is the bigop/real-order grind required to *discharge* it.

**Consequence for the lever:** because difficulty piles onto C, and C is operator_lemmas' target, **the lever scales with difficulty.** The hardest proofs are where lemma-finding hurts most and where the tool pays the most. This is the single most important structural finding in the audit: tool ROI is *increasing* in difficulty.

## 4. Tool Verdict

### `operator_lemmas` (lemma-finding C) — pays off across every kind and every difficulty. Ship it; it is the load-bearing tool.

It is the named tool in the dominant-cost frontier of **17 of 20 lemmas**, across every kind: bigop assembly (PIR_correct, Bad_bound, Pr_PIR_s), FMap/eq_except (pr_Game0_Game1, Plog_Psample, BR93_correct), measure/distribution (D4_Sample, D4_D6, the four phoare-1/2 proofs), group/ring/field (schnorr ×2, pedersen, cpa_ddh0), and — critically — **project-local and clone-qualified** lemmas where name-guessing fails (`D4_6.pr_sampleWi`, `is_restr_*`, `GP.ZModE.ZModpRing.subr_eq0`, same-file `pr_newbad`/`negBadE`/`fI`/`fK`/`OW_OWr`). The three lemmas where it is *not* dominant — sxor2_cons (pure induction, no named lemmas), and the two near-pure-B coupling frontiers — are exactly the cases with little C to find. **Verdict: operator_lemmas is correctly prioritized; difficulty-scaling makes it pay more, not less, on hard proofs.**

### `invariant-frame` / call-skeleton — helps ONLY on identical-program live-var equality; marginal-to-misleading otherwise.

The data draws a sharp line:
- **Where it works (the live-var-equality case):** identical-program self-equivs and abstract-adversary calls with no extra state. `call (: true)` (pedersen, cpa_ddh0 D.distinguish), `call (_: ={glob Log, glob LRO})` (pr_Game2_OW a1/a2), the self-byequiv `={glob OnBound.Gi(D,LRO), arg} ==> ={glob, res}` (G0_G1_G2), `while {1} (={ret})` (Sample_lossless). The audit calls these "the case where a relate-live-vars skeleton actually helps."
- **Where it is marginal:** any invariant that is a *bound* not an equality — Bad_bound's counting invariant, the `while (n<=qP /\ card<=qF)`. The skeleton can offer the live-var *list* (white-given) but not the predicate.
- **Where it actively MISLEADS:** the membership-indicator and big-xor invariants where `s1 != s2`. Pr_PIR_s is explicit: "a relate-the-live-vars skeleton **actively misleads** (s1 != s2)." Same for PIR_correct.

**Verdict:** invariant-frame is a low-value, narrowly-scoped scaffold. It should be *gated to identical-program equivs* and suppressed (or its live-var output explicitly labelled "equality only") where the two programs build different state, or it will hand the agent a wrong predicate on exactly the hardest proofs.

### Still-unmet mechanical need: **structural navigation — indices, swap offsets, and the deterministic frontier-reaching sequence.**

This is C (mechanical, derivable from the two programs) but operator_lemmas does **not** touch it, and the audit flags it as a real, repeated, tedious cost:

- **Swap offsets dominate alignment proofs.** `swap{1} 3 -2. swap{1} 4 -2.` (Plog_Psample), `swap{1} 9 -3. swap{1} 11 -4. swap{2} 5 2.` (pr_Game0_Game1), `swap{1} 7 -5.` (cpa_ddh0 — "read off neither program in isolation"), `swap{2} 12 -5.` (schnorr SHVZK), and es0_Gb's five-swap cascade where the file comment itself says "need complex swap alignment." pr_Game2_OW's audit: "Choosing the exact swap offsets... is the tedious-but-derivable bookkeeping that **dominates this proof**."
- **rcond / statement indices after inlining.** `rcondf 17` (BR93_correct), `rcondt{1} 25`, `rcondf{1} 28` (schnorr SHVZK), `seq 3:` (Bad_bound). The deadness is B; *the index* is C-navigation — "the index 17 is mechanical-once-inlined."
- **The frontier-reaching sequence** `proc; inline*; wp; sp` is A when trivial but becomes real counting work in long inlined bodies.

No shipped tool computes these from the two inlined programs. Today the agent hand-counts statement positions — error-prone and repeated on every alignment-heavy proof (which is most game-hops and sampling equivs).

A second, smaller unmet need: **phoare/seq tactic-form scaffolds.** `tactic_forms` is named for the tricky shapes — the `seq k: post p1 p2 p3 p4 inv` numeric schedule (Bad_bound), `exists*/elim*` ghosting, two-argument `rnd (f) (finv)`, `while true (variant) k (eps)` argument order (Sample_lossless: "mis-ordering these args is the likely failure mode"). This is real but lower-volume than navigation.

## 5. The Next Lever

**Build a program-alignment / navigation engine that computes swap offsets, rcond/seq indices, and the inline-and-reach sequence from the two inlined programs.**

Grounding in the audit's recurring C-needs:
- It is the **single most tedious part** of pr_Game2_OW ("dominates this proof"), the explicit blocker comment in es0_Gb, and a named cost in pr_Game0_Game1, cpa_ddh0, Plog_Psample, schnorr SHVZK, and BR93_correct — i.e. **8 of 20 lemmas**, concentrated in the game-hop / sampling-equiv kinds that make up the bulk of crypto proofs.
- It is **purely mechanical** (the audit classes every swap-offset frontier as C, "computable from the two programs") — so, like operator_lemmas, it removes grind without touching the agent's B job.
- It is **complementary to operator_lemmas**: operator_lemmas owns lemma-finding C; this owns navigation C. Together they cover essentially all of C, leaving the agent only A (free) and B (its actual job).

Concretely, the engine should, given the inlined LHS/RHS: (a) emit candidate `swap{i} k off` sequences that bring nominated samplings/calls into pairwise-adjacent alignment; (b) resolve `rcondt/rcondf <pos>` and `seq <k>` indices by counting post-inline statements (with the agent supplying the *which-branch-is-dead* B-insight and the tool supplying the *index*); (c) surface the `proc; inline *; wp; sp` reach-sequence to a named frontier. A close second priority is folding the **phoare/seq numeric-schedule and two-argument-`rnd` forms** into `tactic_forms` so the argument order is never guessed.

Net: operator_lemmas already captures the largest C slice and **scales up with difficulty**; the alignment/navigation engine captures the next-largest C slice that is currently entirely on the agent. Build it next.
